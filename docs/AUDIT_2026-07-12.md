# Audible And Efficiency Audit

Date: 2026-07-12

> Historical audit input. The P0 findings in sections A, E4, and T1 are now
> implemented and regression-tested; use `docs/ai_context/CURRENT_STATE.md` and
> `docs/ROADMAP.md` for current status.
> P1 spatial scoring, beam search, immutable caches, shared phase diagnostics,
> and the one-command runner are also implemented.

## Headline

The optimizer already has strong harm-prevention: null masking, driver-share
checks, asymmetric-EQ penalties, conservative filter spaces, exact-objective
refinement, phase-reference validation, candidate-only writes, and AFPX linting.
The next gains should not come from more APF sophistication, another optimizer
library, a larger language model, or longer prompts.

The highest-value work is:

1. repair three active-objective integrity gaps;
2. score tonal changes across positions instead of only the centre trace;
3. search combinations more systematically with the existing exact objective;
4. remove repeated local work and emit one tiny assistant-facing result.

These changes run locally and require no model/API credits during optimization.

## Confirmed Strengths

- One authoritative scalar objective drives search and coordinate refinement.
- The mono sum is not the only evidence: solo traces support L/R and driver-share
  decisions.
- Destructive pair-summing regions are excluded from tonal reward, and boost in
  those regions is penalized.
- Frequency grids are normalized once to 96 points per octave.
- Candidate generation is data-guided and passband-aware rather than blind
  random EQ guessing.
- Polarity, delay, and APF follow an evidence-gated ladder and write only to new
  candidate files.
- Compact context routing and local JSON summaries already avoid much repeated
  conversation review.

## Audible-Correctness Findings

### A1. Peak-vs-dip asymmetry is missing from the active objective

The public description says peaks cost more than dips, and the fallback scorer
has a positive-deviation term. The authoritative external objective computes a
symmetric RMS plus absolute worst deviation; it never computes a separate
positive-error term. Therefore a +4 dB peak and -4 dB dip of equal width receive
roughly equal tonal treatment.

Impact: high. This changes which filters survive, especially in the presence
band. Fix before tuning any other objective weights.

### A2. L/R balance can hide frequency-varying mismatch

The authoritative objective ERB-smooths `left-right` but reduces it to a signed
median, then applies `abs()` to that one number. Positive and negative imbalance
across frequency can cancel. The correct report needs both:

- signed broad bias, which predicts image pull;
- weighted RMS or median absolute mismatch, which catches changing imbalance.

Impact: high. L/R imaging was the original reason the mono-only objective was
rejected, so this metric must not contain a cancellation loophole.

### A3. Named tonal components are currently aliases

When the external objective is active, `_optimizer.py` maps `tonal_masked` into
`tonal_error_db`, `sum_tonal_anchor_db`, `presence_error_db`, and
`pareto_tonal_db`. Family selection then treats those aliases as independent
metrics. Reports look more informative than the underlying calculation, and
balanced/aggressive picks effectively count one tonal number several times.

Impact: medium-high. It does not change the best scalar-objective candidate, but
it can change which family files are presented as safest/balanced/aggressive.

### A4. Spatial measurements are loaded but not used by normal PEQ scoring

`position_id` is parsed, and specialist midbass code knows left-ear/right-ear
filenames, but the authoritative objective scores one central measurement set.
That leaves the optimizer vulnerable to a correction that wins at the centre
while degrading both ears.

Impact: very high once consistent three-position measurements are available.
Use centre-weighted median/percentile scoring, not a simple average that can hide
one bad seat position.

### A5. Measurement-condition warnings do not gate optimization

The manifest detects changed source volume and timing references, but the main
optimizer does not consume that manifest. It can continue into tonal/L/R scoring
with incompatible levels. Timing-only measurements can tolerate level changes
after local normalization; tonal and driver-share calculations cannot.

Impact: high harm-prevention. Add explicit tonal-session and phase-session
validity modes rather than one blanket pass/fail.

### A6. PEQ and phase writes are optimized independently

The crossover ladder is calculated from the measured baseline, then the same
phase plan is attached to every PEQ candidate. A candidate PEQ near a crossover
can change level and biquad phase enough to invalidate the baseline phase
solution, but the current magnitude-only objective cannot see that interaction.

Impact: high for combined PEQ+phase candidates. The first safe fix is a guard:
when a phase write exists, reject PEQ candidates that materially alter either
branch in the crossover band. Full complex biquad prediction can come later.

### A7. Sub filters do not receive all front-channel guardrails

Sub changes affect predicted system tone, but front-band count, added-band
guardrails, and headroom loops primarily cover `CH_KEYS` (front channels). Safe
mode prevents sub boosts, which limits harm, but wasted sub cuts and parsimony
are not represented consistently.

Impact: medium. Include sub in applicable parsimony, inert-band, and cascade
checks without pretending one sub measurement describes two unequal outputs.

## Search And Local-Compute Findings

On the historical dense REW test set after 96-ppo normalization:

- grid: 1,054 points;
- authoritative score throughput: about 17.4 candidates/second/process;
- guided pool: 55 data-derived bands across seven groups.

### E1. Guided pools are still combined mainly by stochastic sampling

The app does the intelligent work of deriving plausible fixes, then samples
their combinations. A deterministic beam stage can evaluate the best partial
combinations with the exact scalar, retain interactions, and seed the existing
coordinate refinement. This should produce better candidates per CPU-minute
without introducing another objective or optimizer dependency.

### E2. Constant baseline work is recomputed per candidate

Baseline cascades, smoothed solo references, driver-share totals, and several
band masks can be cached after `_init()`. The objective currently rebuilds some
of them repeatedly. Cache only immutable arrays and verify bit-equivalent scores.

### E3. Phase diagnostics repeat at worker checkpoints and merge

Each stream worker runs crossover diagnostics while writing checkpoints/final
outputs, and merge runs them again. A 20-minute, many-worker run can repeat the
same phase grid search hundreds of times. Compute a fingerprinted phase audit
once per input session, or apply phase writes only during final merge.

### E4. Internal objective precision is rounded too early

The authoritative objective returns a scalar rounded to 0.001. Reports should be
rounded; optimizer comparisons should retain full precision so hardware-step
refinement and tie-breaking are deterministic.

## Token And Workflow Findings

One sampled worker summary measured:

- full `optimizer_summary.json`: 15,563 bytes;
- native payload with one candidate: 3,885 bytes;
- actual assistant decision core: about 1,394 bytes.

### T1. Emit `assistant_summary.json` directly

Write a stable, bounded artifact containing only input fingerprints, validation,
baseline-vs-best named deltas, three family files, phase actions/warnings, and
re-measure instructions. Assistants should not need to transform the full native
summary on every run.

### T2. Add one synchronous local orchestration command

One command should perform manifest validation, choose bounded workers, run,
merge, verify family files, and print only the path to `assistant_summary.json`.
This reduces model tool calls, status questions, and accidental duplicate runs.

### T3. Consolidate competing phase entry points

The active ladder plus three specialist phase scripts overlap. Keep specialist
models as explicit experimental engines, but route normal work through one
canonical API and one report schema. This reduces contradictory recommendations
and the amount of code an assistant must inspect.

### T4. De-duplicate `_tunefit.py` carefully

The root and objective copies are about 1,340 lines each and differ by only a
small patch. Moving to one importable DSP module would nearly halve the largest
repeated context surface. Do this after objective regression fixtures exist;
portability is not worth an untested scoring drift.

### T5. Declare dependencies and supported entry point

There is no normal requirements/packaging manifest. A minimal locked dependency
file and one supported launcher reduce environment-debug turns without changing
DSP behavior.

## Audible Features Worth Building Later

### Spatially robust PEQ

Highest objective improvement after A1-A6. Apply each candidate's exact filter
delta to centre/left/right system and pair traces, then score centre-weighted
median plus a high-percentile/worst-position term. Require improvement to hold
across positions for narrow/asymmetric corrections.

### Same-level sub blend recommendation

Sub level is highly audible, but raw output-level writing should wait for
confirmed AFPX gain semantics and same-level measurements. First emit a
recommendation-only trim with headroom and crossover-summation checks. Do not
simulate it with a broad PEQ boost.

### Explicit voicing audition set

The supplied target remains authoritative. Optionally generate small,
user-labelled target variants such as reference, slightly warmer, and slightly
less presence, then optimize each independently. Never claim measurements can
choose the listener's preferred voicing, and never silently alter the target.

### Complex PEQ/crossover prediction

Model the complete complex RBJ peaking transfer function and only enable it when
solo-to-together validation passes. This lets PEQ and delay/polarity/APF be
evaluated together. Until then, keep the crossover interaction guard.

## Do Not Prioritize

- Bigger language models or cloud/API optimization.
- Arbitrary +4 to +5 dB broad-LF boost exceptions.
- More APF search sophistication before objective/spatial fixes.
- A GUI or live analyzer before the engine has regression fixtures.
- Blind crossover-frequency/slope writing.
- An optimizer zoo; beam plus the current exact objective and refinement is
  enough to test first.
- Machine learning trained on one vehicle or one microphone position.

## Success Standard

Every roadmap item must demonstrate at least one of:

- a named objective invariant fixed by a regression test;
- better worst-position behavior on a held-out multi-position session;
- a better exact objective at equal CPU time without weaker guardrails;
- fewer local passes/tool calls or a materially smaller assistant artifact;
- byte-scoped AFPX/PCT6 writes verified by round trip.

No item is complete merely because it adds code or produces a lower opaque
number.
